# 长对话记忆压缩方案

## 1. 文档目标

本文档给出一套**适配当前项目**的长对话上下文管理方案，目标不是泛泛介绍 memory，而是提供可以直接落地到现有 `Streamlit + ReactAgent + LangGraph Agent` 架构中的设计。

本方案聚焦解决的问题：

1. 用户多轮长对话后，历史消息不断累积，导致 prompt 过长
2. 现有系统只保留最近若干轮，容易忘掉更早但仍然重要的信息
3. 不同类型的信息价值不同，不能简单按时间截断

本文档默认不替代现有的文档上下文压缩能力，而是在其基础上新增一层**对话历史记忆压缩**。

---

## 2. 当前项目现状

### 2.1 当前对话历史如何进入模型

当前前端会把完整聊天记录保存在 `st.session_state.messages` 中，并在每轮提问时传给 `ReactAgent.execute(...)`：

- `app.py` 负责存储和回传对话历史
- `tools/reactagent.py` 负责将历史转换为 `HumanMessage / AIMessage`
- `tools/mid_ware.py` 负责切换主提示词与报告模式提示词

当前历史处理逻辑的核心特点：

1. 只保留最近 `N` 轮
2. 不做摘要
3. 不提取长期事实
4. 不按 token 预算控制

这意味着当前方案属于**短期窗口型 memory**，适合短对话，但不适合持续多轮任务型交互。

### 2.2 当前方案的问题

当用户持续对话时，会遇到以下问题：

- 早期但重要的信息被最近轮次窗口截断
- 用户偏好、长期约束、已确认决策无法稳定保留
- 历史消息越多，模型输入 token 越高
- 如果工具返回长文本，历史膨胀会更快

典型表现：

- 用户一开始说“以后都用中文回答”，后面容易丢失
- 用户连续多轮讨论一个技术方案，前面确认过的边界条件被忘掉
- 多轮调试中，已做过的排查步骤无法稳定复用

---

## 3. 设计目标

### 3.1 核心目标

设计一套**分层式对话记忆系统**，同时满足：

1. 保留最近对话的连贯性
2. 保留长期有效的重要信息
3. 避免历史消息无限增长
4. 尽量少改现有调用链

### 3.2 量化指标

| 指标 | 当前现状 | 目标值 | 测量方式 |
|------|----------|--------|----------|
| 单轮历史输入 token | 随轮次线性增长 | 稳定在 2000-4000 | 请求前估算 |
| 长期事实保留率 | 低 | >= 90% | 人工测试集 |
| 多轮任务连续性 | 中 | 明显提升 | 任务型评测 |
| 额外延迟 | 0 | 摘要触发时 <= 1 次 LLM 调用 | 实测 |
| 接入复杂度 | 低 | 最小侵入式修改 | 文件改动数 |

### 3.3 非目标

本方案第一阶段不追求：

- 完整的长期向量记忆检索系统
- 跨会话持久化记忆
- 用户画像数据库
- 多租户 memory 服务

第一版只解决**单个会话内的长对话上下文过长**问题。

## 4. 总体设计

## 4.1 分层记忆架构

采用三层记忆模型：

```text
┌──────────────────────────────────────────────┐
│                用户当前问题                  │
└──────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────┐
│ Layer 1: Recent Window                       │
│ 最近 6~8 轮原始消息，保留对话连续性          │
└──────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────┐
│ Layer 2: Conversation Summary                │
│ 较老对话的滚动摘要，保留背景和关键过程       │
└──────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────┐
│ Layer 3: Structured Facts                    │
│ 长期有效的结构化事实：偏好、约束、决策       │
└──────────────────────────────────────────────┘
```

三层分别解决不同问题：

- `Recent Window`：解决“当前轮次上下文连续性”
- `Conversation Summary`：解决“较老历史不应全部丢失”
- `Structured Facts`：解决“长期有效信息要稳定保留”

### 4.2 与文档上下文压缩的关系

需要明确区分两类压缩：

| 类型 | 压缩对象 | 当前项目模块 | 目标 |
|------|----------|-------------|------|
| 文档上下文压缩 | 检索出的 `Document` | `rag/context_compressor.py` | 缩短参考资料 |
| 对话记忆压缩 | 聊天历史消息 | 新增 memory 模块 | 缩短会话历史 |

二者可以同时存在，建议最终 prompt 结构如下：

```text
[系统提示词]
[结构化记忆]
[会话摘要]
[最近几轮原始历史]
[当前用户问题]
[必要时工具返回内容]
```

## 5. 数据结构设计

### 5.1 原始历史消息

保留现有结构，不破坏前端展示逻辑：

```python
{
    "role": "user" | "assistant",
    "content": "消息文本"
}
```

这部分仍由 `st.session_state.messages` 保存，用于 UI 回显。

### 5.2 摘要记忆结构

建议新增：

```python
st.session_state.conversation_summary = {
    "summary_text": "",
    "covered_message_count": 0,
    "last_update_turn": 0,
}
```

字段说明：

- `summary_text`：已经压缩好的历史摘要
- `covered_message_count`：摘要已经覆盖了前多少条原始消息
- `last_update_turn`：最近一次摘要发生在第几轮

作用：

- 避免每次都重做全文摘要
- 支持增量摘要
- 便于判断哪些消息还没有被吸收到摘要中

### 5.3 结构化记忆结构

建议新增：

```python
st.session_state.memory_facts = {
    "user_profile": {
        "name": None,
        "language": "zh",
        "preferences": []
    },
    "task_state": {
        "current_goal": None,
        "repo": None,
        "constraints": [],
        "decisions": [],
        "open_questions": []
    }
}
```

#### 字段含义

`user_profile`

- `name`：用户自述姓名
- `language`：长期输出语言偏好
- `preferences`：风格偏好、格式偏好、禁止项等

`task_state`

- `current_goal`：当前任务目标
- `repo`：当前关注的仓库或路径
- `constraints`：不可修改项、限制条件、工具限制等
- `decisions`：已确认技术决策
- `open_questions`：尚未解决的问题

### 5.4 运行时组装结构

建议在进入 `ReactAgent.execute()` 前拼装出一个中间结构：

```python
PreparedConversationContext = {
    "memory_summary": "...",
    "memory_facts_text": "...",
    "recent_messages": [...],
}
```

然后由 `ReactAgent` 使用这份结构去构造最终发送给模型的消息列表。

## 6. 触发策略设计

### 6.1 为什么不能每轮都摘要

如果每轮都摘要，会有三个问题：

1. 成本高：每轮额外增加一次 LLM 调用
2. 延迟高：对普通短对话不划算
3. 摘要漂移：过于频繁重写摘要，容易引入噪声

因此建议使用**阈值触发**。

### 6.2 推荐触发条件

#### 条件 1：轮数触发

当总对话轮数超过阈值时触发。

推荐值：

- `summary_trigger_turns = 12`
- `recent_turns = 6`

解释：

- 总轮数超过 12 轮时，说明对话已经开始变长
- 仅保留最近 6 轮原文
- 更早历史进入摘要层

#### 条件 2：token 触发

当原始历史消息粗估 token 超过阈值时触发。

推荐值：

- `max_history_tokens_before_summary = 3000`

估算方式可以先用粗估：

```python
estimated_tokens = total_chars // 2
```

如果后续接入 `tiktoken` 或模型官方 tokenizer，可替换为精确估算。

#### 条件 3：增量触发

第一次摘要完成后，如果又新增了若干轮新消息，再触发增量摘要。

推荐值：

- `summary_increment_turns = 4`

即：摘要后每新增 4 轮，再把较老部分并入摘要。

### 6.3 推荐默认配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `recent_turns` | 6 | 保留最近原始轮次 |
| `summary_trigger_turns` | 12 | 超过此轮数开始摘要 |
| `summary_increment_turns` | 4 | 摘要后每新增多少轮再做增量摘要 |
| `max_history_tokens_before_summary` | 3000 | 历史 token 上限 |
| `memory_enable_structured_facts` | true | 是否启用结构化记忆 |

## 7. 摘要策略设计

### 7.1 摘要目标

对话摘要不是普通“总结一下”，而是要服务未来对话，因此摘要必须保留：

1. 用户长期偏好
2. 当前任务目标
3. 已确认约束
4. 已做出的关键决策
5. 尚未解决的问题
6. 未来很可能再次引用的上下文

### 7.2 不应保留的内容

以下信息默认不应进入摘要：

- 寒暄
- 重复确认
- 不含新信息的长回复
- 已解决且后续不会再引用的临时细节
- 单次闲聊内容

### 7.3 推荐摘要 prompt

```text
请将以下历史对话压缩为后续多轮对话可复用的记忆摘要。

保留：
1. 用户偏好与长期设定
2. 当前任务目标
3. 已确认的约束条件
4. 已做出的关键决策
5. 尚未解决的问题
6. 对未来回答仍然重要的信息

不要保留：
1. 寒暄
2. 重复表达
3. 与当前主任务无关的闲聊
4. 已解决且不再需要的局部细节

请使用简洁结构化要点输出。
```

### 7.4 推荐摘要输出格式

建议固定成类似如下格式：

```text
- 用户偏好：
  - 使用中文回答
  - 偏好可落地、具体的方案

- 当前任务：
  - 为当前 RAG Agent 设计长对话记忆压缩方案

- 已确认约束：
  - 不混淆文档上下文压缩与会话记忆压缩
  - 保持当前 Streamlit + ReactAgent 架构

- 已做决策：
  - 短期窗口 + 摘要记忆 + 结构化事实三层记忆

- 未解决问题：
  - 还需具体实现 memory manager 和 prompt 拼装逻辑
```

这样比自然语言大段总结更稳定，也更方便后续二次摘要。

## 8. 结构化记忆提取策略

### 8.1 为什么要加结构化记忆

仅靠自然语言摘要，容易出现：

- 关键信息被遗漏
- 长时间后摘要逐渐漂移
- 同类信息重复出现

结构化记忆的作用是把真正长期有效的信息稳定保留下来。

### 8.2 建议提取的事实类型

#### 用户长期偏好

- 输出语言
- 风格偏好
- 是否偏好简洁
- 是否偏好文档化、结构化回答

#### 当前任务状态

- 当前任务目标
- 关注的模块/文件
- 已确定的技术方向
- 已知限制条件

#### 关键决策

- 已采纳的方案
- 明确拒绝的方案
- 关键实现边界

#### 未完成问题

- 后续还要继续追问的点
- 当前尚未解决的技术问题

### 8.3 不建议提取的内容

- 临时表达
- 无复用价值的中间描述
- 助手普通解释性文字
- 单次工具输出细节

### 8.4 更新策略

推荐两种做法：

#### 做法 A：随摘要一起更新

每次触发摘要时，同时从新一批历史里抽取结构化事实。

优点：

- 逻辑简单
- 更新时机统一

缺点：

- 粒度较粗

#### 做法 B：每轮轻量提取

每轮对用户新消息做轻量规则提取，如：

- “以后都用中文” → 更新 `language`
- “不要改 xxx 文件” → 写入 `constraints`

优点：

- 更及时
- 对关键偏好保留更稳

缺点：

- 实现稍复杂

建议第一版采用**做法 A 为主，做法 B 只处理少量高价值规则**。

## 9. 发送给模型的消息拼装顺序

### 9.1 推荐顺序

最终输入模型的内容建议按以下顺序拼装：

1. 系统提示词
2. 结构化记忆
3. 对话摘要
4. 最近若干轮原始消息
5. 当前用户输入

### 9.2 为什么这样排序

- 系统提示词放最前，保持角色和规则优先级最高
- 结构化记忆比摘要更稳定，放在摘要前
- 摘要用于提供历史背景
- 最近原始消息用于保持当前对话的自然连贯性
- 当前用户输入放最后，让模型聚焦当前问题

### 9.3 推荐文本形式

建议把结构化记忆和摘要分别包装成单独的文本块：

#### 结构化记忆文本

```text
【长期记忆】
- 用户语言偏好：中文
- 当前任务：设计长对话记忆压缩方案
- 当前约束：
  - 保持现有 ReactAgent 结构
  - 优先做可落地设计
- 已确认决策：
  - 使用 recent window + summary + facts
```

#### 摘要文本

```text
【历史摘要】
- 之前讨论了文档上下文压缩与会话历史压缩的区别
- 当前项目已有 short_term_turns 截断机制
- 需要补充长对话 memory 设计
```

### 9.4 推荐注入方式

第一版推荐作为额外消息插入，而不是重写 middleware。

理由：

- 对现有 `dynamic_prompt` 机制侵入更小
- 逻辑更集中在 memory manager
- 便于单独测试

## 10. 模块设计建议

### 10.1 新增模块

建议新增：

```text
memory/
├── conversation_memory.py
└── prompts.py
```

也可以放在 `utils/` 或 `rag/` 下，但推荐单独建 `memory/`，职责更清晰。

### 10.2 核心类设计

建议设计一个 `ConversationMemoryManager`：

```python
class ConversationMemoryManager:
    def __init__(self, llm=None, config=None):
        ...

    def should_compact(self, messages: list[dict], summary_state: dict) -> bool:
        ...

    def update_summary(
        self,
        messages: list[dict],
        summary_state: dict,
    ) -> dict:
        ...

    def extract_facts(
        self,
        messages: list[dict],
        memory_facts: dict,
    ) -> dict:
        ...

    def build_memory_blocks(
        self,
        summary_state: dict,
        memory_facts: dict,
    ) -> list[str]:
        ...

    def prepare_messages_for_model(
        self,
        conversation_history: list[dict],
        current_user_input: str,
        summary_state: dict,
        memory_facts: dict,
        recent_turns: int,
    ) -> list:
        ...
```

### 10.3 各方法职责

`should_compact`

- 判断是否触发摘要
- 基于轮数和 token 双阈值

`update_summary`

- 对较老历史做滚动摘要
- 维护 `covered_message_count`

`extract_facts`

- 从摘要输入或历史消息中更新结构化记忆

`build_memory_blocks`

- 把结构化记忆和摘要转换成可拼接到 prompt 的文本块

`prepare_messages_for_model`

- 统一负责最终模型输入消息拼装

## 11. 与现有代码的接入点

### 11.1 `app.py`

`app.py` 是最合适的 session_state 管理位置。

建议新增维护：

- `st.session_state.conversation_summary`
- `st.session_state.memory_facts`

在每轮请求前：

1. 判断是否需要更新摘要
2. 判断是否需要更新结构化记忆
3. 将结果交给 `ReactAgent.execute(...)`

### 11.2 `tools/reactagent.py`

当前 `ReactAgent.execute(...)` 的职责是：

- 接收 `conversation_history`
- 做最近轮次裁剪
- 组装消息
- 调 agent stream

建议保留这个职责，不要把复杂 memory 逻辑全部塞进这里。

更推荐的方式是：

- `app.py` 或 `memory manager` 先准备好 memory block
- `ReactAgent` 只负责把这些 block 插入消息列表

### 11.3 `tools/mid_ware.py`

当前 middleware 只做：

- 调用前后日志
- 主提示词 / 报告模式提示词切换

第一版不建议把 memory 逻辑放进 middleware。

原因：

- memory 更像输入构造问题，而不是 prompt 选择问题
- 放在 middleware 不利于单测
- 会让职责边界变模糊

## 12. 推荐落地步骤

### Phase 1：最小可用版本（推荐先做）

目标：先实现“最近窗口 + 滚动摘要”。

内容：

1. 新增 `ConversationMemoryManager`
2. 在 `session_state` 中增加摘要状态
3. 请求前按阈值判断是否摘要
4. 模型输入时拼接：
   - 摘要
   - 最近 6 轮
   - 当前问题

这一版就能显著改善长对话问题。

### Phase 2：结构化记忆

内容：

1. 增加 `memory_facts`
2. 在摘要更新时同步抽取关键事实
3. 把 facts 作为单独 block 加到模型输入中

这一版能提升稳定性，减少摘要漂移。

### Phase 3：轻量规则提取

内容：

1. 对部分高价值信息做规则级提取
2. 如语言偏好、禁止项、当前任务目标等

这一版能让关键偏好更快生效。

### Phase 4：可选的检索式历史记忆

当会话极长、主题很多时，再做：

1. 将历史消息分片向量化
2. 根据当前 query 检索相关历史
3. 与 recent window 组合使用

此项不建议第一版就做。

## 13. 测试设计

### 13.1 基础功能测试

建议新增测试：

1. 超过阈值后能触发摘要
2. 未超过阈值时不摘要
3. 摘要后 recent window 仍然保留
4. 结构化记忆能正确更新

### 13.2 连续性测试

构造场景：

1. 用户前面说“以后都用中文回答”
2. 中间插入多轮普通讨论
3. 后面再次追问
4. 检查系统是否仍然使用中文

### 13.3 任务型测试

构造长任务会话：

1. 用户先描述目标
2. 中间多轮修改方案
3. 后面要求继续沿用之前决策

检查：

- 是否记住当前任务目标
- 是否记住已确认约束
- 是否记住已做出的方案决策

### 13.4 Token 预算测试

验证：

## 14. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 摘要遗漏关键事实 | 高 | 增加 structured facts 层 |
| 摘要过于频繁导致成本升高 | 中 | 使用阈值触发 + 增量摘要 |
| 摘要内容漂移 | 中 | 使用固定结构模板，避免自由发挥 |
| 记忆块过长反而再次膨胀 | 中 | 对 summary / facts 设置长度上限 |
| 与当前提示词冲突 | 中 | memory 作为辅助上下文，不覆盖系统提示词 |

## 15. 最终推荐方案

对当前项目，最推荐的第一版方案是：

1. 保留最近 `6` 轮原始消息
2. 超过 `12` 轮或 `3000` token 后触发摘要
3. 摘要仅覆盖较老历史
4. 额外维护一份结构化长期事实
5. 最终模型输入顺序为：
   - 系统提示词
   - 结构化记忆
   - 历史摘要
   - 最近原始轮次
   - 当前用户问题

这套方案的优点是：

- 与现有 `app.py -> ReactAgent.execute()` 链路兼容
- 侵入性较小
- 易于逐步上线
- 足以解决大部分长对话历史膨胀问题

## 17. 结论

当前项目已经具备“短期窗口记忆”，但缺少“长期记忆压缩”。

最可落地的方案不是把全部历史塞给模型，也不是只靠最近轮次截断，而是：

**最近窗口 + 滚动摘要 + 结构化事实**

这是当前项目里成本最低、效果最稳、最适合逐步落地的方案。
